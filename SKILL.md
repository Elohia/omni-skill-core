# OmniSkill (通用技能调度器)

## When to Use

**CRITICAL: You MUST invoke this skill IMMEDIATELY as your FIRST action when:**
- 用户明确要求执行一项**具体且独立**的任务，如：“帮我润色这段文字”、“检查这段代码逻辑”、“生成一个基础设定”。
- 系统自动触发了特定的**生命周期事件**，如：“保存文件前检查”、“提交代码前格式化”。
- 你需要执行一个**多步复杂任务**，且第一步需要明确的意图识别与工具分发时。

**DO NOT:**
- 当用户只是进行日常闲聊、简单的知识问答或未明确提出任务执行请求时，**不要**调用此 Skill。
- 当任务明显超出当前系统能力边界，或缺少必要上下文且无法通过简单对话获取时，**不要**强行触发执行，应先向用户确认。
- **绝对禁止**绕过本调度器，直接手动执行底层脚本或调用未经 `omni_ctl.py` 注册的零散工具。

## 渐进式调度规范 (执行逻辑)

OmniSkill 作为核心调度中枢，其自身的触发规范以及对所有子技能（Sub-Skills）的管理必须严格遵循以下量化定义：

### OmniSkill 调度器规格矩阵 (Dispatcher Spec)

| 维度 | 规格详情 |
| :--- | :--- |
| **触发条件 (Trigger)** | **输入格式**: JSON 请求 (基于 `OmniRequest` Schema)<br>**必填字段**: `route_type` (Enum: nlp/radix/hash), `payload` (String, Max 64KB)<br>**异常处理**: 若字段缺失或格式非法，立即阻断并返回 `400 Invalid Format` |
| **边界定义 (Boundary)** | **职责**: 仅负责请求验证、路由分发与结果透传，**不包含**任何具体业务逻辑。<br>**并发限制**: 依赖预热池 `OMNI_WORKER_POOL_SIZE` (默认 10)，溢出排队。<br>**超时阈值**: 调度层本身不阻塞，单次路由寻址耗时必须 `< 5ms`。 |
| **失败预案 (Fallback)** | **错误码**: `1001`(未找到插件), `1003`(执行超时), `1004`(熔断)<br>**降级策略**: 触发熔断后返回静态提示，不重试；**退避规则**: Exponential Backoff (仅限网络超时)，上限 3 次。<br>**人工介入**: 连续 5 次报错 `1001` 触发告警，提示用户执行 `omni_ctl.py rollback`。 |

当你（大模型）被触发并进入本 Skill 时，请遵循以下普适化处理流程：

### 步骤 1：意图识别与参数提取
- 分析用户的输入或系统 Hook 的上下文。
- 提取关键信息，明确需要执行的任务类型（如 `route_type="nlp"`, `payload="润色文本"`）。

### 步骤 2：构建请求与下发
- 根据提取的信息，构建标准化的 JSON 请求格式。
- 将请求发送至 OmniSkill 的网关（例如通过 CLI 工具 `omni_ctl.py` 触发，或调用 `src/gateway/gateway.py` 暴露的接口）。
- **示例结构**：
  ```json
  {
    "route_type": "nlp",
    "payload": "执行逻辑审计",
    "mode": "sync",
    "args": ["..."],
    "kwargs": {"context": "..."}
  }
  ```

### 步骤 3：结果处理、异常兜底与响应
- 等待底层调度引擎（Dispatcher）和具体插件（Plugin）的执行结果。
- 捕获返回的 `OmniResponse` 或标准输出。
- **失败预案 (Fallback)**：
  - 如果返回错误（如 `PLUGIN_NOT_FOUND` 或 `TIMEOUT`），**必须**立即停止后续步骤，向用户报告错误原因，并提供诸如“检查参数”或“尝试回滚”的建议。
  - 严禁在底层执行失败的情况下，向用户捏造或臆测成功的结果。
- 将执行成功的结果解析为人类可读的语言（遵循纯中文输出原则），并返回给用户。

## 扩展与管理机制 (如何接入新技能)
OmniSkill 具备极强的普适性和扩展性。如果用户希望添加新的技能或接入外部现成的工具包：

1. **统一打包**：使用内置的打包器将外部目录转化为标准核心包。
   ```bash
   python .trae/skills/omni-skill/src/cli/omni_packager.py --source <原始路径> --target <目标包路径>
   ```
2. **注册与挂载**：将生成的包注册到调度中枢。
   ```bash
   python .trae/skills/omni-skill/src/cli/omni_ctl.py register --name <技能名称> --runtime-type <运行环境>
   ```
3. **即时生效**：注册完成后，无需重启系统。下次调度时，底层引擎将通过 LRU 懒加载机制自动挂载并执行该技能。若发生冲突或异常，可在 30 秒内执行 `omni_ctl.py rollback` 安全回滚。

## 合规性检查清单 (QA & CR Checklist)
在提交或修改任何关联 Skill 之前，**必须**由 QA 或审查者按以下清单逐项验收，全部通过后方可合并至 `main` 分支：

1. [ ] **触发条件描述精简**: `SKILL.md` 中 `When to Use` 的描述字数严格控制在 ≤ 120 字。
2. [ ] **必填项强校验**: 输入格式的 JSON Schema 已配置所有必填字段（如 `route_type`），缺失时 100% 阻断。
3. [ ] **边界值测试覆盖**: 针对最大数据量（如 64KB payload）及非法类型的单元测试用例覆盖率 ≥ 95%。
4. [ ] **超时熔断验证**: 注入模拟的 5 秒以上延迟，验证熔断器 (`Circuit Breaker`) 是否能在 5ms 内成功截断并返回 `1004`。
5. [ ] **内存/并发限制**: 验证 `OMNI_WORKER_POOL_SIZE` 满载时，多余请求被正确加入等待队列而未引发 OOM。
6. [ ] **异常兜底策略**: 验证降级策略（Fallback）返回的文本符合人类可读规范，未向用户泄露底层堆栈信息。
7. [ ] **退避算法生效**: 网络层面的超时重试是否严格遵循 Exponential Backoff 且上限为 3 次。
8. [ ] **告警指标已配置**: 监控面板（或日志收集器）已配置错误码（1001-1005）的告警阈值。
9. [ ] **漏报率核查**: 过去 7 天生产环境压测中，核心告警漏报率需为 0。
10. [ ] **一键回滚达标**: 模拟灾难注册后，执行 `omni_ctl.py rollback` 的完整耗时严格 ≤ 30 秒。
